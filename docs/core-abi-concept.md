# Core ABI Concept

OpenAGI treats ABI as a runtime pattern, not a company-specific product.

The core loop is:

```text
Integrations -> Signals -> Workflows -> Directional Adaptive Scrutiny
-> Agent Layer -> Multi-Tiered Memory -> Custom Context + Propagation
-> Outputs -> Feedback Loop
```

## Directional Adaptive Scrutiny

Scrutiny is the system's selection pressure. It evaluates what is happening, why it matters, what criteria should apply, and whether the system should act, ask, watch, ignore, or propagate.

It should be directional, but not outcome-locked. The system can know what it is trying to improve without assuming the answer in advance.

## Memory

Memory is split into tiers:

- Short-term: immediate working context.
- Medium-term: repeated operational context.
- Long-term, or Lava: compressed intuition and durable truths.

The runtime should remember repetitive or novel/high-risk information, preserve critical specifics, and compress or forget low-value detail.

## Propagation

Propagation is specialization through division. A specialist should be created only when a task is repeated enough or novel/risky enough to justify a bounded worker with its own scope, memory, success metric, and tool access.

The goal is not more agents. The goal is less generalized burden.

## Integrations

Integrations are just input adapters. They can come from files, APIs, browser state, calendars, codebases, MCP tools, user feedback, or external systems.

No single integration is the core product.
